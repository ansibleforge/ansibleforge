DEMO 1

IIS Auto-Heal

A web server health check that fixes itself in under a minute.

The Scenario

"IIS just started returning 500s on /health.aspx. Page someone."

Without EDA

  • Monitoring fires a PagerDuty alert
  • Engineer wakes up, RDPs in
  • Runs iisreset
  • Closes the ticket, goes back to bed
  • Average MTTR: 15–30 minutes

With EDA

  • Rulebook polls /health.aspx every 10s
  • 4xx/5xx response triggers a rule
  • Rule launches the restart job template
  • Counter increments, audit trail in AAP
  • Average MTTR: ~45 seconds

The Rulebook

extensions/eda/rulebooks/iis-health-restart.yml 


- name: IIS Auto-Heal
  hosts: all
  sources:
    - ansible.eda.url_check:
        urls:
          - "{{ IIS_HEALTH_URL }}"
        delay: 10
  rules:
    - name: IIS unhealthy -> restart
      condition: >-
        event.url_check.status == "down" or
        event.url_check.status_code >= 400
      throttle:
        once_within: 1 minute
        group_by_attributes:
          - event.url_check.url
      action:
        run_job_template:
          name: "IIS Demo - Restart IIS"
          organization: Default

Throttle prevents storms during the heal window. URL is templated from extra_var so the same rulebook works against any host.

What the Action Does

Job Template IIS Demo - Restart IIS — playbooks/iis-demo/restart.yml

Add IIS host to runtime inventory — pulls host + admin password from the IIS Demo Secrets credential
Remove fail.flag — the file health.aspx checks for to decide its status code
Increment restart counter — C:\inetpub\wwwroot\restartcount.txt
Run iisreset — reload IIS to pick up any underlying config changes

Reusable: launch it from the AAP UI on demand, from EDA, or from a scheduled workflow.

The Chaos Generator

A second activation drives a continuous failure-and-recovery loop — you don't have to break things by hand to demo. 


- name: IIS Chaos
  hosts: all
  sources:
    - ansible.eda.generic:
        payload:
          - chaos: tick
        loop_count: -1
        loop_delay: 90        # base interval
  rules:
    - name: Inject IIS error
      condition: event.chaos == "tick"
      action:
        run_job_template:
          name: "IIS Demo - Chaos Tick"
          # adds 0-30s jitter, then writes fail.flag

Tick every 90–120 seconds; toggle the activation on/off to start/stop the demo.

Live: What to Watch

1. Browser

  • Open /health.aspx
  • Refresh during a chaos tick → 500
  • Wait ~30s → 200 again

2. AAP — EDA

  • Activations → IIS Auto-Heal
  • Rule Audit shows each fire
  • Status & events processed counter ticks

3. AAP — Jobs

  • IIS Demo - Restart IIS in history
  • Each run takes ~25s
  • Output shows restart count: N

Talking point: point at the restart counter going up — that's the audit trail your auditors want.

DEMO 2

ServiceNow VM Provision

A self-service catalog that actually self-services.

The Scenario

"Users want a VM. Today they file a ticket and wait two days for someone to click in the AWS console."

Without EDA

  • RITM lands in the cloud team queue
  • Someone reads the variables
  • Logs into AWS, picks subnet/SG/AMI
  • Launches, then updates the RITM by hand
  • Creates the CMDB CI by hand

With EDA

  • RITM hits ServiceNow, EDA polls every 30s
  • Rule fires the provision job template
  • Playbook reads RITM vars, launches EC2
  • Closes the RITM with work notes
  • Creates the CMDB CI — same flow for decom

The Rulebook

extensions/eda/rulebooks/snow-vm-provision.yml — two rulesets in one file 


- name: Provision VM on ServiceNow catalog request
  sources:
    - servicenow.itsm.records:
        instance: { host: "{{ SN_HOST }}", username: "{{ SN_USERNAME }}",
                    password: "{{ SN_PASSWORD }}" }
        table: sc_req_item
        sysparm_query: "cat_item=2c160bb0...^state=1"
        interval: 30
  rules:
    - name: New VM request -> launch provisioning
      condition: event.sys_id is defined
      throttle: { once_within: 1 minute, group_by_attributes: [event.sys_id] }
      action:
        run_job_template:
          name: "VM Provision - EC2"
          job_args:
            extra_vars:
              ritm_number: "{{ event.number }}"
              ritm_sys_id: "{{ event.sys_id }}"

A second ruleset in the same file watches the decommission catalog item and fires VM Deprovision - EC2.

What the Provision JT Does

playbooks/vm-provision/provision_ec2.yml

Fetch RITM variables — servicenow.itsm.api_info pulls vm_name, OS, cores, memory, disk, env
Auto-discover networking — first VPC + public subnet + shared vm-provision-sg (SSH/HTTP/HTTPS/RDP/WinRM)
Resolve AMI — map OS family to owner+filter, pick newest creation date
Resolve instance type — cores_memory map → t3a.medium, r5a.xlarge, etc.
Launch EC2 — tagged with the RITM number for traceability
Create CMDB CI — cmdb_ci_vm_instance with the instance ID + IP
Close the RITM — PATCH state=closed_complete with work notes

The Chaos Generator

Same pattern as the IIS chaos: a generic-source activation creates work for the real activation to react to. 


- name: SN Chaos
  sources:
    - ansible.eda.generic:
        payload: [ {chaos: tick} ]
        loop_count: -1
        loop_delay: 300       # ~5 min base
  rules:
    - name: Submit provision + decommission RITMs
      condition: event.chaos == "tick"
      action:
        run_job_template:
          name: "SN Chaos Tick"
          # 0-180s jitter, then:
          #  1) POST a new provision RITM via /servicecatalog/order_now
          #  2) Find existing chaos VMs in CMDB and submit decom RITMs
5–8mCycle
1VM alive at a time
~30sProvision
~15sDecom

Live: What to Watch

1. ServiceNow

  • RITM list — new ones land every cycle
  • State transitions to Closed Complete
  • Work notes show instance ID + IPs
  • CMDB CIs created & retired

2. AAP — Jobs

  • SN Chaos Tick every 5–8 min
  • VM Provision - EC2 right after
  • VM Deprovision - EC2 for previous VM
  • Drill into stdout for live AWS calls

3. AWS Console

  • EC2 list filtered by tag:Name=chaos-vm-*
  • One instance running, rotating
  • Terminated instances visible (1h)
  • Tag ServiceNow_RITM proves the link

Talking point: the RITM and the EC2 instance carry the same correlation ID — full audit, both sides.

Patterns Worth Stealing

The Chaos Generator

  • Second activation with ansible.eda.generic
  • Drives the demo without manual breakage
  • Toggle is_enabled to start/stop
  • Random jitter in the action playbook keeps cycles natural

Throttle is Mandatory

  • once_within + group_by_attributes
  • Stops 6 polls in 60s firing 6 remediations
  • Group by the entity key (URL, sys_id)
  • Test by triggering 3 events at once

Restart Policy: always

  • Activation pod dies → auto-restart
  • Watermarks reset on restart — expected
  • Pair with health metrics in Grafana

Source Watermarks

  • servicenow.itsm.records only emits new records since startup
  • "Activation is silent" usually means "no new events"
  • Use log_level: info when troubleshooting
